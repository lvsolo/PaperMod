# 论文精读文档写作规范（Paper Reading Writing Spec）

> 用途：把本仓库 `content/posts/paper_reading/` 下「论文精读」类 markdown 文档的所有写作要求汇总成一份独立规范。开新的 Claude Code 会话时，**直接把本文件作为输入让 Claude 读完**，即可快速对齐全套要求，无需重新说明。
>
> 适用范围：`content/posts/paper_reading/**/*.md`（典型示例：`FlashAttention.md`、`Sparse4D.md`、`VLA/ViT.md`、`VLA/DINO.md` 等）。

---

## 0. 一句话总纲

**中文为主、英文术语保留；原理要有官方代码佐证；流程图用 Mermaid、公式用真 LaTeX；结构对齐 FlashAttention 范式；体现该篇在技术路线中的位置。**

---

## 1. 文件与标题约定

- **一篇论文一个文件**，文件名用论文常用简称（如 `ViT.md`、`CLIP.md`、`DecisionTransformer.md`、`DiffusionPolicy.md`、`OpenVLA.md`）。
- 系列论文可合并（如 `Sparse4D.md` 用表格列 v1/v2/v3）。
- 文档**标题（H1 / frontmatter title）必须与论文一一对应**，能让人一眼看出是哪篇。
- 如果存在总览/路线图文档（如 `VLA/VLA_guideline.md`），需在其中**为每篇精读文档加跳转链接**（相对路径，如 `🔗 精读笔记: [ViT.md](ViT.md)`）。

---

## 2. 文档结构（对齐 `FlashAttention.md` 范式）

每篇精读文档**按此骨架**组织（章节标题可据论文调整，但顺序与精神保持一致）：

1. **Hugo frontmatter**（见 §3）
2. `# 论文信息` —— 标题 / 作者 / 机构 / 发表会议 / arXiv 链接 / 官方代码链接（都用列表）
3. `> **一句话总结**` —— 一段加粗的引用块，高度浓缩核心贡献
4. `# 1. 引言/背景与动机` —— 要解决什么问题、之前方法的不足、本文的切入点
5. `# 2. 方法` —— 核心方法，分小节讲清楚；**每个原理小节后配官方代码片段**（见 §6）
6. `# 3. 实验` —— 关键实验结论，用表格/对比呈现
7. `# 4. 局限与讨论`（可选）
8. `# 5. 核心要点总结` —— 用 Mermaid 总结图 + 3~5 条 takeaway
9. `# 6. 参考资料` —— 原论文 arXiv、官方代码、关键相关工作（带链接）

---

## 3. Frontmatter（Hugo / PaperMod）

固定字段：

```yaml
---
title: "论文英文标题 (常用简称)"
author: "lvsolo"
date: "YYYY-MM-DD"
tags: ["paper reading", "<领域标签1>", "<领域标签2>", ...]
ShowToc: true
TocOpen: true
---
```

- `author` 固定 `lvsolo`。
- `tags` 第一项固定 `paper reading`，其余按论文主题（如 `attention`、`VLA`、`contrastive learning`）。
- `ShowToc: true` / `TocOpen: true` 让目录默认展开。

---

## 4. 语言与写作风格

- **中文为主，关键术语保留英文**（如 `patch embedding`、`self-attention`、`covariate shift`），首次出现可「中文（English）」并列。
- 面向「工程导向、想真正读懂论文」的读者：重直觉、重数据流、重「为什么这么做」。
- 多用**对比**（与同类方法）、**类比**（把抽象概念类比成具体事物）、**分点**，避免大段平铺。
- 适度使用加粗 `**...**` 强调关键结论；引用块 `>` 放重要洞察。

---

## 5. 图表：用 Mermaid，不要 ASCII

- **流程图 / 架构图 / pipeline / 训练阶段 / 数据流 / 模块关系图 / 关系树**：一律用 ` ```mermaid ` 代码块（站点已配置自动渲染），**禁止用 `┌─┐│ ▼ ←` 这类 ASCII 画框图**（会错行、难读）。
  - 首选 `flowchart TD`（纵向）或 `flowchart LR`（横向）；有分组用 `subgraph`。
  - **节点文本必须用双引号包起来**（含中文/特殊字符），节点内换行用 `<br/>`。
  - 例：
    ```
    ```mermaid
    flowchart TD
        A["原始图像 H×W×C"] --> B["切 patch + 展平"]
        B --> C["Transformer Encoder"]
    ```
    ```
- **保留为普通代码块 / 文字 / 表格**（不要强行转 mermaid，反而更糟）的内容：
  - 数据表格 → 用 markdown 表格（不要 ASCII 对齐表格）。
  - 公式推导、伪代码步骤（带 `← 注释` 的步骤说明）。
  - 纯文字对比框、要点列表。

> 站点已支持 Mermaid 自动渲染（检测到 `mermaid` 代码块即加载），无需额外配置。

---

## 6. 数学公式：用真 LaTeX，不要"伪公式"

- **所有数学表达式用真正的 LaTeX**，让它在 Markor / 站点上都能渲染：
  - 行间公式：`$$...$$`（独占一段）。
  - 行内符号：`$...$`。
- **禁止**把数学式子当成普通文本写进 ``` 代码块（例如 `x_t`、`S_ij`、`O(N²)` 这种用下划线当下标、用 Unicode 当上下标的写法）——下划线会原样显示，极难看。
- LaTeX 写法要点：
  - 下标 `x_{t}`、`S_{ij}`；上标 `N^{2}`、`Q^{\top}`。
  - 求和 `\sum_{i=1}^{N}`、乘积 `\prod_{t=1}^{L}`、分数 `\frac{a}{b}`、根号 `\sqrt{d}`、范数 `\|\cdot\|`、期望 `\mathbb{E}`、实数集 `\mathbb{R}`。
  - 希腊字母 `\sigma \epsilon \alpha \tau \lambda \Sigma`；运算 `\cdot \times \to \langle\cdot,\cdot\rangle`；函数名 `\text{softmax}\,\log\,\max`。
  - 矩阵转置 `^{\top}`；分段函数用 `\begin{cases}...\end{cases}`。
- 渲染基础设施已就绪：站点 `config.yaml` 已开 `params.math: true`（全站 KaTeX），`layouts/partials/extend_head.html` 已配置 `$...$`(行内)/`$$...$$`(行间) 的 KaTeX auto-render。Markor 自带 KaTeX 可直接渲染。

---

## 7. 原理配官方代码（关键要求）

- **在描述算法原理的小节后面，贴出对应「官方（或最权威）代码库」的真实实现片段**，并用**中文行内注释**把代码逐行映射回原理。
- 目的：让自然语言描述的原理落到具体实现上，加深理解（参考 `VLA/DINO.md` 的 `DINOLoss.forward`、`DataAugmentationDINO`、`get_last_selfattention` 三段代码）。
- 取材优先级：
  1. 论文官方代码库（如 `facebookresearch/dino`、`haotian-liu/LLaVA`、`openvla/openvla`）。
  2. 若官方是 JAX / 闭源 → 取**最权威的 PyTorch 实现**（如 ViT 取 `timm`、CLIP 损失去 `open_clip`）。
  3. 若完全闭源（如 PaLI / RT-1 / RT-2）→ 用开源复现或**清晰标注的伪代码**，并注明"官方未开源"。
- 片段要**聚焦**：每个原理配 1~2 个关键函数即可，不要整文件粘贴。保留原有变量名，删去与本原理无关的分支（用 `# ...` 注释说明删节）。
- 代码块用 ` ```python `（站点有语法高亮）。

---

## 8. 体现"在技术路线中的位置"

- 每篇文末应点明：**这篇在整个学习路线 / 技术演进里处于什么位置、启发了谁、被谁继承**。
- 例：ViT 是 CLIP/DINOv2/SigLIP/OpenVLA 视觉骨干的基础；Decision Transformer 启发了 RT-2/OpenVLA 的"action 当 token 自回归"。
- 可用一张 Mermaid 关系图收尾（如 OpenVLA 文末把 Vision→VLM→Action→VLA 整条路线收束）。

---

## 9. 内容质量底线

- **不要敷衍**：原理要讲到能回答"输入输出是什么、token/特征怎么设计的、loss 是什么、为什么这么做"。
- 数据/结论尽量给具体数字与对比（表格优先）。
- 关键设计要给**直觉解释**，而不只是罗列步骤。
- 保持客观，区分"论文声称"与"后续社区验证/改进"。
- **概念/术语首现必释（重要）**：正文里出现的每一个专业概念、缩写、评测指标（如 `k-NN`、`linear probing`、`covariate shift`、`EMA`、`centering/sharpening`、`InfoNCE`、`return-to-go`、`action tokenization`……），都要在**首次出现处**用一两句话解释清楚"它在测什么 / 它是什么意思 / 为什么要它"——不要假设读者已经懂。最好是配合一个最小流程图或公式。读者应该是"看完这篇就能跟别人讲明白这个术语"的程度。典型范例：`VLA/DINO.md §4.2.1` 对 k-NN 的解释、`VLA/ViT.md §2.4.1` 对 pos_embed 的解释。
- **自检**：每写完一个术语，问一句"一个没接触过这个概念的人，看这段能不能 get 到它在干嘛？"答不上来就补。
